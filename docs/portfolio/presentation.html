제품 정의

수명이 짧고 반복 제작이 잦은 이벤트·모집 페이지를 비개발 직군이 10분 내에 배포할 수 있는 셀프서비스 랜딩 빌더. 템플릿 선택 → 폼 편집 → URL 배포 → UTM 트래킹까지 하나의 워크플로우로 제공한다.

v1.0 목표

  • 템플릿 기반 10분 내 페이지 배포
  • 만들기 → 배포 → 채널 공유 → 유입 분석 end-to-end 연결
  • 마케터·운영 담당자의 반복 제작 비용·시간 절감
  • Free / Pro / Business 요금 체계 수립

01   BACKGROUND

왜 이 제품이 필요한가

이벤트 안내, 모집 공고, 세일 프로모션, 공지 페이지는 수명이 짧고 자주 반복된다. 그런데 기존 대안은 각각 다른 문제를 갖고 있었다.

01

외주·개발 요청

긴급 이벤트에 맞지 않는 긴 리드타임. 문구 하나 바꿀 때도 다시 요청해야 함.

02

미리캔버스 · 피그마

디자인은 가능하지만 배포·호스팅·UTM 트래킹이 분리되어 마케터 워크플로우가 끊김.

03

노션 · 구글사이트

빠르게 올릴 수 있지만 CTA·이벤트 섹션·채널별 유입 분석에 최적화되지 않음.

→ 빨리 만들고, 바로 공유하고, 효과를 확인하고, 다음에 또 빠르게.

Quickpage 3 / 17
02   TARGET

누구를 위한 제품인가

Primary · 마케터
캠페인 랜딩이 필요한 마케터

인스타·뉴스레터·광고 등 채널별로 다른 UTM 링크가 필요하고, 배포 후 유입 데이터를 확인해야 한다. 디자인 툴을 깊게 다루기보다 빠른 배포와 트래킹을 원한다.

Primary · 운영 담당자
이벤트·공지를 자주 올리는 운영자

행사 안내 페이지를 당일 수정·재배포해야 한다. 개발팀에 요청하지 않고 스스로 페이지를 관리하고 싶다.

03   HYPOTHESIS

제품 가설

가설내용검증 방법
H1 · 배포 속도 비개발 직군도 템플릿만으로 10분 내 페이지를 배포할 수 있다 편집기 UX 테스트 + 첫 배포율 측정
H2 · 트래킹 가치 UTM 방문 로그가 마케터에게 실질적 가치를 제공한다 트래킹 링크 생성 + 대시보드 제공
H3 · 유료 전환 워터마크 제거·분석·도메인으로 Free→Pro 전환이 가능하다 요금제 설계 + Phase 2 결제 연동

North Star Metric — 주간 활성 배포 페이지 수 (Weekly Published Pages). 가입 수나 매출보다 먼저, 페이지가 실제로 배포·공유되는 행동을 본다.

포지셔닝

범용 웹사이트 빌더가 아닌,
「이벤트·모집 랜딩을 10분에 배포」에만 집중.

  • 미리캔버스 → 디자인 자유도 vs 배포·UTM 분리
  • 노션 → 빠른 공유 vs 랜딩·CTA 약함
  • 외주 → 완전 커스텀 vs 비용·시간

Quickpage = 빠른 제작 + 랜딩 목적 최적화 + 트래킹 내장

05   CORE FEATURES

v1.0 기능 명세 — 개요

Golden Path 기준 3개 핵심 기능으로 분류. 각 기능별 상세 명세는 UI/UX · 시나리오 · 예외 · 비즈니스 로직으로 작성.

05-1
페이지 편집·배포
/templates → /create → /p/[id]

템플릿 기반 폼 편집, 실시간 미리보기, 로그인 후 URL 배포. v1.0 핵심 가치인 「10분 배포」를 담당.

05-2
페이지 관리
/dashboard

배포된 페이지 목록·폴더·워크스페이스·휴지통 관리. 수정·이름 변경·삭제·복구 등 운영자 워크플로우.

05-3
방문 분석·트래킹
/p/[id] → /dashboard

UTM 파라미터 수집, 채널별 트래킹 링크 생성, 방문 로그 대시보드. 마케터의 캠페인 성과 확인 니즈 대응.

05-1   FEATURE SPEC

페이지 편집·배포

/create · /create?edit=[id]
UI / UX Interface
  • 좌측(또는 우측) 설정 패널 + 실시간 미리보기 2분할
  • MO / PC 미리보기 토글, 설정 패널 좌우 전환
  • 섹션: 레이아웃 · 콘텐츠 · 디자인 · 고급(커스텀 CSS)
  • 상단 「배포하기」 CTA — 신규 / 수정 배포 구분
  • 비로그인 배포 시 LoginPromptModal (alert 미사용)
User Scenario
  • 템플릿 선택 후 /create 진입
  • 할인율·기간·혜택·CTA 입력, 레이아웃·색상 조정
  • MO 미리보기로 최종 확인
  • 배포하기 → 로그인(필요 시) → /p/{id} 이동
  • 기존 페이지는 /create?edit={id}로 재진입·수정 배포
Exception
  • 비로그인 + 배포 클릭 → 모달, 편집 내용 유지
  • API 401 → 로그인 유도, 배포 중단
  • 배포 API 실패 → 에러 메시지 노출, 재시도 가능
  • Supabase 미연결 → 상단 경고 배너, 로컬 데모 모드
  • 페이지 로딩 중 → 로딩 상태 표시
Business Logic
  • 페이지 데이터는 JSON 스키마로 저장, 단일 렌더러로 출력
  • 신규 POST /api/pages, 수정 PUT /api/pages/{id}
  • 배포 시점에만 로그인 필수 — 편집·미리보기는 비로그인 허용
  • 배포 성공 시 UUID 기반 /p/{id} URL 즉시 발급
  • 활성 워크스페이스 ID와 함께 저장

페이지 관리

/dashboard
UI / UX Interface
  • 사이드바: 개인·워크스페이스 전환, 폴더, 휴지통
  • 필터: 전체 · 즐겨찾기 · 폴더별
  • 페이지 카드 — 배너·제목·수정일·즐겨찾기
  • 카드 ⋮ 메뉴: 열기·이름 변경·방문 로그·휴지통·추출

User Scenario
  • 배포 후 대시보드에서 페이지 목록 확인
  • 폴더 생성 → 캠페인별 페이지 분류
  • 카드 메뉴 → 수정(/create?edit) 또는 공개 페이지 열기
  • 불필요 페이지 → 휴지통 이동 → 7일 내 복구 또는 영구 삭제

  • 미로그인 접근 → 로그인 페이지 리다이렉트
  • 타인 페이지 → API 권한 검증으로 접근 차단
Business Logic
  • 삭제는 소프트 삭제(deletedAt), 휴지통 7일 보관
  • 워크스페이스·폴더 단위로 페이지 스코프 분리
  • 즐겨찾기·이름 변경은 즉시 반영
  • 휴지통 복구 시 deletedAt null 처리
  • 페이지 소유자만 수정·삭제·분석 접근 가능

05-3   FEATURE SPEC

방문 분석·트래킹

/p/[id] · /dashboard
UI / UX Interface
  • 공개 페이지 — 소유자에게만 트래킹 링크 생성 패널
  • UTM source / medium / campaign 입력 → 링크 복사
  • 대시보드 ⋮ → 「방문 로그」 모달
  • 총 방문 수, 채널별 비율, 최근 방문 이력
  • 방문자에게는 로그·패널 미노출 (프라이버시)
User Scenario
  • 세일 페이지 배포 후 트래킹 링크 생성
  • utm_campaign=summer_sale 로 인스타·광고 채널별 게시
  • 방문자가 UTM 링크로 랜딩 페이지 접속
  • 대시보드 방문 로그에서 채널별 유입 확인
  • 성과 낮은 채널 문구·채널 조정 후 재공유
Exception
  • 비소유자 → 트래킹 패널·로그 접근 불가
  • tracking_events 테이블 미구성 → 설정 안내
  • 방문 0건 → 「아직 방문 기록 없음」 빈 상태
  • UTM 없이 직접 접속 → referrer 기록
  • 봇·중복 방문 → v1.0은 단순 집계 (디듑 미적용)
Business Logic
  • 페이지 로드 시 utm_source/medium/campaign/referrer 수집
  • tracking_events 테이블에 pageId별 이벤트 저장
  • 트래킹 링크 = /p/{id}?utm_* 쿼리 자동 조합
  • 대시보드에서 source·medium·campaign별 집계
  • 분석 데이터는 운영자 전용, 공개 페이지에 미노출

06   KEY DECISIONS

주요 의사결정

v1.0 설계 시 확정한 핵심 결정 사항.

배포 시점에 로그인을 필수로 할 것인가?
예. 익명 배포 허용 대신 배포 시 로그인을 요구. 페이지 소유권·대시보드·트래킹이 한 계정에 묶이도록 설계. 비로그인 사용자는 alert 대신 모달로 전환 유도.
페이지 데이터를 어떻게 저장·렌더링할 것인가?
JSON + 단일 렌더러. 페이지마다 커스텀 HTML 대신 JSON 스키마 하나로 6종 레이아웃을 렌더링. 템플릿 확장·유지보수 비용 최소화.
방문 로그를 어디에 노출할 것인가?
대시보드로 이동. 공개 페이지에 로그 노출 대신 운영자 전용 대시보드에서 확인. 방문자 프라이버시·운영 UX 모두 고려.
