# 회고 (Retrospective)

---

## 프로젝트 목적

- **기획자 포트폴리오**로서 문제 정의 → MVP 범위 → 프로토타입 → 로드맵까지 한 사이클 증명
- 실제 창업보다 **「제품 사고방식 + 실행 검증」** 에 초점

---

## 잘한 점

| 항목 | 내용 |
|------|------|
| **니치 집중** | 범용 빌더가 아닌 「10분 이벤트 랜딩」으로 스코프 축소 |
| **Golden Path** | 만들기 → 배포 → 공유 → 트래킹까지 end-to-end 연결 |
| **범위 관리** | 결제·도메인·팀은 기획만 하고 코드에서 의도적 제외 |
| **UX 디테일** | 로그인 모달, MO/PC 미리보기, 글꼴 드롭다운 등 실사용 고려 |
| **문서화** | PRD·로드맵·지표·데모 스크립트로 「기획만 한 것」과 차별 |

## 아쉬운 점 / 다음에 다르게 할 것

| 항목 | 반성 | 개선 |
|------|------|------|
| 기능 욕심 | 레이아웃 6종·트래킹 등 프로토타입 대비 풍부 | 포트폴리오 초기에 Implemented/Future 표 고정 |

| HTML 추출 | 레이아웃 미반영 | 우선순위 낮추거나 문서에 한계 명시 (완료) |

## 배운 점 (PM 역량)

1. **가설을 기능 목록이 아니라 퍼널로 쓴다** — 배포율, UTM 사용률
2. **프로토타입은 「설득용 최소 완성도」** — 전 기능 구현 ≠ 좋은 포트폴리오
3. **의도적 미구현도 기획 역량** — Phase 2 문서가 「다음에 뭘 할지 안다」는 증거
4. **전환 마찰 설계** — 배포 시점 로그인, 모달 톤, 편집 draft 유지
